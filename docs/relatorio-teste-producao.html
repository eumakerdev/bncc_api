01

O que é esta API?

A BNCC (Base Nacional Comum Curricular) é o documento oficial que define o que cada aluno deve aprender em cada ano da escola, da creche ao ensino médio. Ela é enorme e vive em PDFs difíceis de consultar.

Esta API funciona como um "garçom digital": um programa faz um pedido (“me traga as habilidades de matemática do 4º ano”) e a API devolve a resposta já organizada, num formato que outros programas entendem — o JSON, basicamente uma lista de informações etiquetadas.

Um mini-dicionário

TermoEm português claro
RequisiçãoO pedido que enviamos para a API.
RespostaO que a API devolveu.
EndpointUm “balcão de atendimento” específico — um só de habilidades, outro só de competências.
HTTP 200Código de sucesso: “deu certo”.
HTTP 400 / 401 / 404Códigos de recusa: “seu pedido tem um problema” (vistos na seção 4).
API keyUma “senha de acesso” pessoal. Sem ela, a API não entrega os dados.
HabilidadeUma competência específica que o aluno deve desenvolver. Cada uma tem um código.

Como ler um código de habilidade

EF 04 MA 10
EF → Ensino Fundamental  ·  04 → 4º ano  ·  MA → Matemática  ·  10 → décima habilidade da lista
02

Como conseguir acesso

Antes de pedir qualquer dado, é preciso ter uma API key (a senha de acesso). O caminho é o mesmo que um desenvolvedor de verdade faria — em quatro passos.

Observação: hoje o envio de e-mail está em modo “console” — o link de confirmação não chega numa caixa de entrada real, aparece apenas nos registros internos do servidor. Para o público se cadastrar sozinho, será preciso ativar o envio de e-mail (SMTP).

1

Criar conta

Enviamos
POST /api/v1/auth/signup
{
  "email": "teste@example.com",
  "password": "********"
}
Recebemos
{
  "account_id": "38bf3540-…",
  "email": "teste@example.com",
  "email_verified": false
}
➜
Conta criada, mas ainda não verificada (email_verified: false).
2

Confirmar o e-mail

Enviamos
POST /api/v1/auth/verify-email
{
  "token": "5hRTphqc…"
}
Recebemos
{
  "email_verified": true
}
➜
E-mail confirmado. Agora a conta pode gerar chaves.
3

Login e gerar a API key

Enviamos
POST /api/v1/keys
{
  "name": "smoke-test"
}
Recebemos
{
  "id": "dfdcf87a-…",
  "name": "smoke-test",
  "key": "bncc_live_zUWBz9Q2…"
}
➜
A chave completa só aparece uma vez. Depois, o sistema guarda apenas uma versão embaralhada, por segurança. Neste teste, a chave foi apagada ao final.

A partir daqui, todo pedido de dados leva a chave no cabeçalho Authorization: Bearer <sua-chave> — como carimbar cada pedido com sua senha de acesso.

E quando o pedido está errado?

Uma boa API precisa recusar com clareza pedidos inválidos, sem quebrar. Testamos três erros de propósito — e a recusa (pill âmbar) aqui é o comportamento correto.

Teste 8

HTTP 400
Enviamos
GET /api/v1/habilidades/CODIGO_INEXISTENTE
Recebemos
{
  "detail": "Código malformado.
     Formatos aceitos: EI, EF, EM…"
}
➜
Em bom português: “Esse código nem tem o formato certo.”
Teste 9

Pedido sem a chave de acesso

HTTP 401
Enviamos
GET /api/v1/habilidades
(sem Authorization)
Recebemos
{
  "detail": "API key ausente."
}
➜
Em bom português: “Você não se identificou.” — a proteção funciona.
Teste 10

Filtro com valor que não existe

HTTP 400
Enviamos
GET /api/v1/habilidades?etapa=nao_existe
Recebemos
{
  "error_code": "validation_error",
  "errors": [{
    "campo": "query.etapa",
    "msg": "deve ser educacao_infantil,
      ensino_fundamental ou ensino_medio"
  }]
}
➜
Em bom português: “Essa etapa não existe; use uma destas.” Todos os erros vêm com mensagem explicativa e sem vazar detalhes internos do servidor.
05

Busca semântica com Inteligência Artificial

O recurso mais avançado. Em vez de filtros exatos, você faz uma pergunta em linguagem natural e a IA (modelo Google Gemini) monta uma resposta baseada nos trechos oficiais mais parecidos em significado — cada achado ganha uma nota de relevância de 0 a 1.

Regra de ouro do sistema: a IA nunca inventa. Se não achar nada confiável, ela avisa. Toda resposta vem marcada como não-oficial e lista as fontes (os códigos oficiais) para você conferir no texto original.

06

Conclusão

✓ O que está funcionando bem

  • Disponibilidade: no ar; respondeu todas as 20 requisições.
  • Velocidade: dados em ~250–290 ms; buscas com IA de 0,4 a 7,7 s.
  • Acentuação correta em tudo (décimos, frações, emoções).
  • Segurança: sem chave bloqueia; erros explicados sem vazar detalhes internos.
  • IA responsável: nunca inventou; admite quando não sabe e sempre cita fontes.

! Pontos a decidir / melhorar

  • Ligações entre dados vazias (Testes 7 e 12): relações e unidades temáticas ainda não populadas.
  • Opção incluir_contexto (S5) não retornou os trechos brutos esperados — confirmar.
  • E-mail em modo “console”: enquanto não houver envio real (SMTP), o público não consegue se cadastrar sozinho.

Metodologia: foi criada uma conta de teste real, verificada e usada para gerar uma API key; a chave foi apagada ao final (confirmação HTTP 204). Todas as requisições e respostas acima são reproduções fiéis do que a API enviou e devolveu em 05/07/2026.
